# Roo Code Prompt Optimizer - Investment Management Edition

## 描述
专门用于优化 Roo Code Architect 模式下的 prompt，充分利用 GPT-5.2 和 high reasoning 设置的能力。
针对**欧洲和亚太地区金融服务公司**的**投资管理领域**解决方案架构设计。

## 使用场景
- 使用 Roo Code Architect 模式进行系统设计
- 模型配置为 GPT-5.2
- Reasoning 级别设置为 High
- **领域**：Investment Management（投资管理）
- **区域**：欧洲和亚太金融市场（非中国国内）
- **角色**：Solution Architect

## Prompt 优化原则

### 1. 结构化思维引导
由于 reasoning 设置为 high，prompt 应当：
- 明确要求分步骤推理
- 提供清晰的问题分解框架
- 鼓励深度思考和多角度分析

### 2. 金融投资管理领域专业性
针对投资管理系统架构设计，prompt 应包含：
- 业务目标和监管合规要求（GDPR, MiFID II, APRA, MAS, SFC等）
- 技术栈偏好和金融行业标准（FIX, SWIFT, ISO 20022等）
- 可扩展性、高可用性（99.99% SLA）和灾备要求
- 安全性、数据隐私和审计追溯能力
- 实时性能要求（市场数据处理、交易执行延迟<100ms等）
- 多区域部署和数据本地化要求

### 3. GPT-5.2 高级推理能力利用
充分利用 GPT-5.2 的高级能力：
- 要求生成完整的架构决策文档和权衡分析
- 期待多种方案对比（技术 + 业务 + 合规维度）
- 利用其强大的金融领域知识和代码理解能力
- 要求深度分析监管合规和风险应对策略

## 实战示例

### 示例 1：投资组合管理系统架构设计

**优化后的 Prompt：**
```
我需要为一家欧洲资产管理公司设计新一代投资组合管理系统，请作为 Solution Architect 进行深度分析：

# 项目背景
- 客户：欧洲 UCITS 基金管理公司
- 规模：€300B AUM，5000+ 投资组合，覆盖股票、固收、衍生品
- 当前：遗留系统（C++），批处理为主，缺乏实时风险监控
- 目标：实时投资组合估值、风险分析（VaR, Stress Testing）、合规监控
- 区域：主站点伦敦，灾备站点法兰克福，需考虑 Brexit 后的数据驻留

# 监管要求
- MiFID II：交易报告、最佳执行、客户资产隔离
- GDPR：客户数据保护、数据本地化
- UCITS：风险限额监控、资产隔离、NAV 计算准确性
- 审计：7 年审计日志保留，不可篡改

# 技术和集成要求
- 技术栈：Java/Spring Boot 或 C#/.NET（团队熟悉）
- 云平台：AWS 或 Azure（欧洲区域）
- 数据供应商：Bloomberg（市场数据、参考数据）、MSCI（风险模型）
- 托管银行：State Street、BNY Mellon（持仓对账、公司行为）
- 交易平台：Tradeweb、Bloomberg EMSX
- 性能：EOD batch < 2小时，实时风险计算 < 30秒，市场数据延迟 < 1秒
- 可用性：99.95% SLA（交易时段 99.99%），RTO 1小时，RPO 15分钟

# 架构设计任务
请进行深度推理和分析：

1. **业务领域建模和服务拆分**
   - 如何划分业务领域？（投资决策、订单管理、组合会计、合规监控等）
   - 微服务 vs 模块化单体？考虑交易原子性和数据一致性
   - 服务边界如何处理跨领域的业务流程（如再平衡交易流程）？
   - 如何设计聚合根和领域事件？

2. **数据架构设计**
   - OLTP vs OLAP 分离策略：交易数据 vs 分析数据
   - 时序数据存储：历史持仓、市场数据、净值曲线（推荐具体数据库）
   - 数据湖策略：监管报告、历史分析、ML 应用
   - 事件溯源：订单生命周期、审计日志的设计
   - 跨区域数据复制和一致性保证

3. **关键业务流程设计**
   - 实时组合估值流程：市场数据更新触发计算
   - 订单管理流程：订单生成 → 合规检查 → 执行 → 清算 → 对账
   - EOD 批处理：NAV 计算、性能归因、风险报告
   - 分布式事务处理：如何保证清算对账的准确性？

4. **非功能性架构**
   - 高可用：主备切换、负载均衡、断路器模式
   - 灾备：跨区域热备、数据同步、灾备演练策略
   - 安全：零信任架构、API网关、加密策略、RBAC
   - 监控：业务指标（NAV偏差、交易延迟）+ 技术指标
   - 性能优化：缓存策略、异步处理、批量计算优化

5. **技术选型建议**
   - 消息中间件：Kafka vs Solace（金融级消息总线）
   - 计算引擎：风险计算用 GPU 加速？Spark 批处理？
   - 时序数据库：InfluxDB vs TimescaleDB vs Kdb+
   - API 设计：RESTful vs GraphQL，如何集成 FIX 协议？

6. **实施和迁移**
   - 如何与遗留系统并行运行？双写策略、数据对账
   - 分阶段实施：MVP（核心估值）→ 风险分析 → 合规报告
   - 监管审批流程和 FCA 沟通策略
   - UAT 测试策略：历史数据回放、性能基准测试

请充分运用你对投资管理行业的理解，提供详细的架构分析和建议。
```

### 示例 2：交易系统性能优化

**优化后的 Prompt：**
```
我们的电子交易平台在市场波动期间性能严重下降，需要系统性优化方案。
请作为 Investment Management Solution Architect 深度分析：

# 系统概况
- 业务：股票、ETF 电子交易执行平台（类似 Bloomberg EMSX）
- 区域：亚太地区（新加坡、香港、东京交易所）
- 技术栈：Java/Spring Boot + PostgreSQL + Redis + Kafka
- 规模：峰值 50,000 订单/小时，200+ 并发交易员
- 监管：MAS 要求交易报告延迟 < 1秒，订单时间戳精度到毫秒

# 性能问题症状
- 正常市场：订单响应 < 100ms ✓
- 市场波动时（如央行决议发布）：
  * 订单响应时间飙升至 5-10 秒 ✗
  * 市场数据处理延迟 > 2 秒 ✗
  * 部分订单超时失败 ✗
  * 交易员抱怨系统"卡顿"

# 现有监控数据
- APM 数据显示：
  * 60% 延迟在数据库查询（持仓检查、合规验证）
  * 25% 延迟在外部服务调用（交易所 FIX gateway、市场数据）
  * 15% 延迟在应用逻辑（风险计算、合规规则引擎）
- 数据库：
  * PostgreSQL 慢查询：持仓查询、订单历史查询有全表扫描
  * 连接池在峰值时耗尽（max 200 connections）
  * CPU 使用率峰值 85%
- Redis：
  * 缓存命中率 70%（期望 95%+）
  * 缓存键设计不合理导致失效过多
- Kafka：
  * 市场数据 topic lag 达到 10 秒
  * Consumer 处理速度跟不上 Producer
```

## 总结

使用此 skill 进行投资管理系统架构设计时，记住：

### 技术层面
- **GPT-5.2** 有强大的推理和代码理解能力，要充分利用
- **High Reasoning** 模式适合复杂的架构决策、多方案权衡和深度分析
- **Architect** 角色需要全局视角、长远思考和跨领域整合能力

### 金融行业层面
- 始终将**监管合规**作为第一优先级，技术方案必须满足合规要求
- 强调**审计追溯性**和数据不可篡改性
- 考虑**多区域部署**的监管和技术复杂性（数据本地化、跨境数据传输）
- 重视**业务连续性**和灾备能力（金融系统对可用性要求极高）
- 理解**金融行业特定约束**（交易时段、清算窗口、零停机要求）
- 重视**供应商尽职调查**和长期技术支持

### Prompt 设计原则
- **上下文越详细，方案越精准**：提供 AUM、资产类别、客户类型、监管要求
- **鼓励推理过程**：要求权衡分析、多方案对比、风险评估
- **结构化提问**：分层次、多维度（技术 + 业务 + 合规）
- **使用行业术语**：帮助模型理解金融投资管理特定场景
- **要求可执行性**：不只要架构图，要实施路径、监管审批、测试策略
- **考虑非功能性需求**：可用性、性能、安全、灾备、可观测性

### 投资管理领域特殊考量
- **数据准确性至关重要**：NAV 计算、持仓对账、公司行为处理
- **实时性和批处理并重**：实时风险监控 + EOD 批处理
- **复杂的集成需求**：数据供应商、托管银行、交易平台、监管报告
- **长周期数据保留**：通常要求 7-10 年审计日志和历史数据
- **跨时区运营**：欧洲、亚太、美洲市场的全球化考虑

通过结构化、专业化、多维度的 prompt 设计，可以最大化发挥 Roo Code + GPT-5.2 在金融投资管理系统架构设计中的价值。
